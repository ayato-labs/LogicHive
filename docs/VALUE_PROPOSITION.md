## 解決する課題：AIがすぐ忘れる「コンテキスト」の喪失

AIエージェント（Cursor / Cline / Claude Code）の進化により、汎用的な関数（S3アップロード、CSVパース等）をゼロから生成するコストは限りなくゼロに近づいている。
しかし、**「AIはあなたの会社のビジネスロジックを知らない」**という根本的な問題は残されたままだ。

「このプロジェクト特有の認証エラー処理」「弊社独自のDBカラム命名規則」「既存のUIコンポーネントとの結合ルール」——これらは汎用データセットには存在せず、AIに毎回ゼロからプロンプトで教え込む必要がある。

| 浪費される資源 | 具体例 |
|:---|:---|
| **コンテキスト伝達の手間** | 「前回作ったあのモジュールの仕様に合わせて」と毎回指示する手間 |
| **レビューコスト** | AIが生成したコードが「自社の作法」に合っているか確認する時間 |
| **一貫性の崩壊** | 毎回微妙に異なるアーキテクチャや依存関係が選定され、技術的負債が溜まる |

> **LogicHiveは、この「失われるコンテキスト」を資産化し、AIエージェントの「長期記憶（ドメイン知識）」として定着させる。**

## 定量的な価値

### 1回の再利用で節約できるもの（汎用生成との比較）

| 指標 | 汎用AI生成（都度指示） | コンテキスト再利用（LogicHive） | 削減率 |
|:---|:---|:---|:---|
| プロンプト作成時間 | 5-10分（前提条件の説明） | 10秒（Search + Get） | **98%** |
| 修正指示の往復 (Turn) | 3-5ターン（仕様のズレ修正） | 0ターン（過去の正解をそのまま注入） | **100%** |
| 結合時のバグ | 「自社の作法」無視による不整合 | なし（自社環境で動作確認済みのロジック） | **100%** |

### 蓄積効果（複利で効く）

```
関数10個蓄積 → 月に 5時間 節約
関数50個蓄積 → 月に 20時間 節約
関数100個蓄積 → もはや「自分のstdlib」
```

蓄積すればするほど、新規プロジェクトの立ち上げが速くなる。
関数100個のライブラリを持つ開発者と、毎回ゼロから書く開発者では、生産性に**数倍の差**が出る。

## 定性的な価値

### 1. 「秘伝のタレ」の永続化

コード自体はAIが書けるが、「なぜそのアーキテクチャを選んだか」「どのライブラリのどのバージョンを使うべきか」という決定事項はLogicHiveに**消えない資産**として残る。
プロジェクトが終わっても、チームメンバーが入れ替わっても、組織の「正解のコードと文脈」は残り続ける。

- 「ウチの決済基盤との連携処理、どう書くのが正解だっけ？」が `search_functions("Stripe integration in our architecture")` で即解決

### 2. 「ゴミ」から「資産」への昇華（コンテキストの構造化）

単なるテキストの保存（Free機能）では、データが増えるほど検索ノイズに埋もれてしまう。
LogicHiveは、保存時に適切なタグや説明文をメタデータとして付与し、高速なSQLベースのキーワード検索エンジンとして機能する。

時間が経つほど、組織内ライブラリ全体の品質が高まり、ストレージ効率も最大化される。

### 3. AIエージェントとの相乗効果

LogicHiveはMCPサーバーとして動作する。
つまり、AIエージェントが**自律的に**関数を検索・取得・活用できる。

```
従来:  人間が過去のコードを探す → コピペ → エージェントに渡す
今後:  エージェントが自分でLogicHiveを検索 → 即座にinstall → 開発継続
```

人間の介在なしに、エージェントが自分の知識ベースを活用する。
これは「AIエージェントに記憶を与える」ことに等しい。

## Before / After

### Before（汎用AI頼りのワークフロー）

```
新しいプロジェクト開始
  → 「社内DBからユーザーデータを引く処理が必要だな」
  → AIに生成させる
  → AIは汎用的なSQLを書くが、社内の論理削除フラグ（is_deleted=1）やテナントIDの絞り込みを忘れる
  → レビューで指摘し、修正させる（往復に15分）
  → 次のプロジェクトでまたAIが同じ仕様を忘れ、同じ指摘をする。
```

### After（LogicHive 導入後）

```
新しいプロジェクト開始
  → AIエージェント: search_functions("fetch users with our internal rules")
  → スコア0.96で "core_user_fetcher" がヒット（論理削除やテナントロジック内包済み）
  → 完璧なコンテキストを持った検証済みコードがプロジェクトに注入される。10秒で完了。
```

## Free（OSSの広場）と Basic（Enterpriseの金庫）の境界

企業が「自社固有のコンテキスト」を保存するツールである以上、LogicHiveのMVPにおける最大の価値は「AIの知能」ではなく**「コンテキストへのアクセス権（データの機密性）」**にある。
すべてのユーザーは「組織（Organization ID）」を持ち、プランによって他の組織とのデータ共有ルールが切り替わる。

- **Freeプラン（Public Bank / SQL Search）**: 保存したコンテキストは**全てMITライセンス**としてLogicHiveのグローバル共有知となり、他組織からも検索可能になる。見返りとして、完全に無料で関数ストレージとして利用できる（LLMは一切関与しないが、高速な**SQLチューニングにより自組織のデータが優先表示される**ため、社内共有ツールとして快適に機能する）。
- **Basicプラン（Private Bank / SQL Search）**: 他組織からのアクセスを完全に遮断し、自社の「秘伝のタレ」を**セキュアに保護（非公開化）**する。「自社のコンテキストを守りながら、AIエージェントに自社専用の知識を注入できる」ことが、たった月額$9で手に入る最大の価値である。

### 🚨 最重要ルール：ライセンスの不可逆性
「とりあえず無料で使ってみて、機密ロジックが含まれていたら後から有料プランにして隠そう」という使い方は**できない**。
無料プラン時に保存されたデータに付与されたMITライセンスは、あとから有料プランへアップグレードした場合でも**絶対に撤回されない（パブリックのまま残る）**。これは、既にそのデータを利用し始めた他の開発者の権利を守り、LogicHiveのオープンなエコシステムを維持するための絶対ルールである。
この不可逆性があるからこそ、機密情報を扱う企業は最初から有料プランを契約する強力な動機となる。

## まとめ

LogicHiveは「ただの便利ツール」ではない。

- **企業・プロジェクト固有の「コンテキスト」を資産化する仕組み**
- **AIエージェントに「ドメイン知識という長期記憶」を与えるインフラ**
- **技術力を証明するポートフォリオ**

この3つの価値が、コストゼロで手に入る。
