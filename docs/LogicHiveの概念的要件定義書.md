# 🛡️ LogicHive 設計思想

## ビジョン
**「過去の優れた実装を二度と書き直さない。良質なロジックをAIエージェントの共有知へ。」**

LogicHiveは、「どのように実装するか」という技術的な解決策ではなく、エンジニアが「どのような思想で自らの資産を管理すべきか」を定義するためのツールです。

---

## 🚫 あきらめるもの（Giving up）
LogicHiveを運用するにあたり、私たちは以下のことをあきらめます。

1. **「記憶」への依存**：
   「以前どこかで書いたはず」という不確かな記憶を頼りにコードを探す、あるいは思い出す時間をあきらめます。
2. **「ゼロ」からのコード生成への期待**：
   AI（Cursor等）が常に完璧なコードをゼロから生成してくれることを期待するのをあきらめます。自分にとっての「正解」は自分の中にしかありません。
3. **「LogicHive側でのLLMによるコード解釈（お節介な加工）」**：
   MCP（LogicHive）側でLLMを使ってソースコードの依存関係（必要なライブラリや環境変数）を抽出しようなどという「善意の加工」をあきらめ、禁じます。**利用側のAIエージェント（CursorやClaude等）が搭載しているLLMの方が常に最新で賢いため、サーバー側で中途半端なメタデータを付与することは「害悪なノイズ」になるからです。** LogicHiveはコードの純粋な保管・提供に徹し、コードの意味解釈はすべて受信側のエージェントの知能に委ねます。
4. **「自動自己修復（Self-Healing）」**：
   環境の変化（依存ライブラリのアップデートや仕様変更等）に合わせてロジックを自動的に修正・修復する機能の実装をあきらめ、禁じます。**本システムを利用するAIエージェントのほうが組み込む知能よりも圧倒的に賢いため、サーバー側で勝手な修復を行うことは、高品質なエージェントの思考を妨げるノイズにしかならないからです。** 劣化の「検知」と「スコアリング」には徹しますが、「修復」はエージェントの知能に委ねます。
5. **「汎用・巨大」なライブラリの肥大化**：
   あらゆるケースに対応した重厚長大なヘルパークラスやライブラリを保守し続ける手間をあきらめます。

5. **UI/UXのプレミアム化（Visual Excellence）**：
   個人で利用するものにそんなものは必要ない。そもそも自分用のツールにUIを豪華にするためにかける時間なんて1msもない。
6. **全言語における自前の静的解析（Syntax Check）**：
   多言語（C++, JavaScriptなど）に対応するための独自パーサーやLinterを自前で保守することをあきらめます。よく使う言語（Python）のみ静的解析で最低限の品質を守り、それ以外の言語はLLMの汎用的な推論能力（厳格なプロンプトによる仮想コンパイラとしての評価）に完全に依存する「割り切り」を採用します。

## ✨ 求めるもの（Seeking）
私たちがLogicHiveを通じて手に入れたいのは、以下の価値です。
ほしいのはきれいなコードではなく、使える有用なコードを保存しておくことで、次の開発の際に楽にするです。

1. **「多言語原子」の再利用**：
   特定の文脈で完璧に動作したロジックだけを「原子」として蓄積し、瞬時に再利用すること。
   python, C, C++, java, html, css etcでも動作可能にする。優先順位は、python > html > css > etcって感じ。理由はpythonとhtml（UI構築）のものを対応しておけば大概の開発が可能だから
2. **エージェントとの「対話的」な適応**：
   AIエージェントにロジックを「提案」させ、それを現状のコードに「適合」させるというプロセス。AIは「生成者」ではなく、自分の資産の「適合者（Adapter）」として扱います。
3. **「知の資産化」による開発体験の向上**：
   書けば書くほど、自分の開発環境が強化される。車輪の再開発を撲滅し、常に「一歩進んだ場所」からスタートできる状態を求めます。
4. **ソフトウェアの「防腐（Anti-Rot）」と価値の維持【システムの肝】**：
   コードは保存した瞬間から環境の変化（ライブラリの更新、言語仕様の変更等）によって劣化（腐敗）が始まります。LogicHiveの真の価値は、保存することではなく、**「そのコードが今も動くのか」を自動で監視し、劣化を検知して信頼スコアを調整する「動的安定化」**にあります。これにより、AIエージェントが「腐ったロジック」を拾うリスクを根絶します。

5. **「レジリエンス（弾力性）」と自己診断 (Self-Diagnostic)**：
   MCPサーバーとの通信における `EOF` エラーやタイムアウトは、開発体験を著しく損なう「沈黙の爆弾」です。LogicHiveは、以下の指針でシステム自体の安定性を維持します。
   - **非同期・非ブロッキング**: 起動時の環境構築や大規模な削除処理を別スレッドに逃がし、サーバーの応答性を常に維持する。
   - **例外の完全封じ込め (Fault Isolation)**: どんな致命的なランタイムエラーが発生しても、サーバープロセスを道連れにせず、適切なエラーメッセージと診断情報（Diagnostics）を返す。
   - **統合診断ツール (`check_integrity`) の提供**: DB、ベクターストア、ランタイムプールの整合性を一括で自己検証できる機能を持ち、ユーザーに不透明な不調を許さない。

5. **究極の到達点：「Auto-Dev（検証・ライブラリ化）機構」**：
   **本質：ただのRAGラッパーが本質で、個人利用においては、venvやdockerはついでの機能**
   LogicHiveの最終形態は単なるスニペット管理ではなく、自律型のAIソフトウェアファクトリーです。この壮大なビジョンに向けて、以下の現実的なロードマップ（漸進的進化）を描きます。
   - **Phase 0 (MVP)**: 同期的なLLM仮想コンパイラによる静的解析（速度優先・現在の実装）
   - **Phase 1 (A案)**: `uv venv`等を用いたローカル実行環境による非同期・動的テスト（Python等の主要言語から対応）
   - **Phase 2 (B案)**: 完全独立したDocker等へのスピンオフによる、全言語対応型の完全隔離動的テスト環境

※主目的は過去の開発資源の正しい再利用のため、AIエージェントの思考リソースの使用量の削減ではないことを明記する。

## 🔄 コア・アルゴリズム（活用フロー）
LogicHiveとAIエージェント（Cursor等）は、以下のサイクルで協調します。技術的な実装方法ではなく、この「フロー」こそがLogicHiveの真髄です。

1. **Request (探索)**：
   開発者がCursor上で「〜する処理が欲しい」とLogicHive MCPにリクエストする。
2. **Retrieve & Inject (抽出と挿入)**：
   LogicHiveが過去の「良質な関数」を検索し、一時的なファイル、あるいはクリップボードに書き出す。
3. **Adaptation (微調整)**：
   Cursorが、書き出された関数を「現在のプロジェクトの変数名や型」に適合するように自動的に微調整する。
4. **Validation (検証)**：
   コードを実行し、うまく動作することを確認する。
5. **Registration (再登録)**：
   微調整を経てさらに洗練された、あるいは新しく生まれた良質な関数を、再びLogicHiveに登録し、資産をアップデートする。

## 品質ゲートの設計原則：「意見」から「論理」への転換

### なぜこの節が存在するのか

LogicHiveの「防腐（Anti-Rot）」機構を構築する過程で、我々は重大な設計上の誤りを犯し、それを是正した。
この節は、その失敗と是正の記録である。同じ過ちを繰り返さないために、**なぜLLMのみによる品質判定が「論理」ではなく「屁理屈（Sophistry）」であったか**を明記する。

### 失敗の経緯：モグラ叩きの3ラウンド

品質ゲート（Quality Gate）の強化において、以下の3段階の試行錯誤を経た。

**Round 1: 「テストを要求すれば良い」**
- 施策: テストコードの提出を必須化し、AIに「テストが十分か」を判定させた。
- 結果: `assert True` のような無意味なテストでも、AIは「テストが存在する」という事実に引きずられ、高得点を与えた。
- 敗因: AIは「テストの存在」と「テストの価値」を区別できなかった。

**Round 2: 「AIを厳しくすれば良い」**
- 施策: AIのペルソナを「Senior Architect」から「Hostile Forensic Auditor」へ変更し、AIスコアが低い場合に最終スコアを強制的に叩き落とす「Veto Power（拒否権）」を導入した。
- 結果: 完全に空のコード（`pass`メソッド）は検出できるようになった。しかし、**ビット演算やメタクラスで「1+1の計算」を過剰に複雑化した偽装コード**に対しては、AIが「ロジック自体は正しい」と判断し、45点という中途半端な温情を与えた。
- 敗因: AIは「技術的正しさ」と「実用的価値」を区別できなかった。

**Round 3: 「閾値をもっと上げれば良い」**
- 施策: 45点以下を強制的に0点に丸める「Score Binarization（二値化）」を計画した。
- 結果: この施策は「なぜ45点ではだめなのか」に対して「閾値を60にしたから」としか答えられない、恣意的なパラメータ調整であった。

**Round 4: 「厳格すぎて使われない」 (ドッグフーディングによる是正)**
- 現象: 品質ゲートが「ブラックボックス」かつ「完璧主義」すぎたため、開発のスピード感が阻害され、ユーザーが登録を躊躇する事態が発生した。
- 敗因: 失敗の理由（Tracebackやstderr）を隠蔽していたこと、および「テストがない＝悪」と決めつけていたこと。
- 教訓: **「厳格な番人」である前に、「親切な整備士」でなければならない。** 失敗の理由を詳細に開示し、かつ完成度に応じた段階的な受理（Draft許容）を行う必要がある。

### 根本原因の特定

3ラウンドの失敗を経て、以下の根本原因を特定した。

**「単一のLLM呼び出しに品質判定を委ねること」自体が、論理的に破綻している。**

| 「論理」に必要な性質 | LLMのみの品質判定 | 判定 |
| :--- | :--- | :--- |
| **再現性** (同じ入力に同じ結果) | 非決定的。同じコードでも実行ごとにスコアが変動する | 不適格 |
| **検証可能性** (第三者が根拠を確認できる) | 「AIがそう言った」以外の根拠がない | 不適格 |
| **決定論性** (数学的・構造的に結果が導かれる) | プロンプトの文言次第で結果が変わる | 不適格 |

つまり、AIのプロンプトをどれだけ工夫しても、ペルソナをどれだけ厳しくしても、それは**「より説得力のある意見」を生成しているだけ**であり、**「誰が見ても否定できない論理的証明」ではない。** 閾値の調整は「意見の大きさ」を変えているだけで、「意見を論理に変える」ことはできない。

### 是正方針：決定論的検証レイヤーの導入

上記の反省を踏まえ、LogicHiveの品質ゲートは以下のハイブリッド・アーキテクチャを採用する。

**原則: 「事実（Fact）」が判定し、「意見（Opinion）」は補助する。**

```
[Layer 1: 決定論的検証 - 事実の層]
  目的: 誰が何度実行しても同じ結果を返す、客観的指標による判定
  手段:
    - **コードの完全性 (Integrity)**: テストコードのAST解析、assert文の有無、空メソッドの検出
    - **セキュリティ監査 (Immune System)**: `eval/exec`の禁止、危険なサブプロセス、ハードコードされたシークレットの静的解析（Rigour由来のロジック）
    - **依存関係の潔白 (No Hallucination)**: `requirements.txt`等に基づき、実在しない外部ライブラリのインポートを拒絶
    - **保守性メトリクス (Maintainability)**: 循環的複雑度や引数過多の客観的計測
    - **実行時検証 (Execution)**: 現在の環境でテストが実際にパスするか
  性質: 再現可能、検証可能、決定論的

[Layer 2: AI補助検証 - 意見の層]
  目的: 決定論では捕捉できない「匂い」（設計の不自然さ、命名の不適切さ等）を検出
  手段:
    - LLMによるセマンティック分析（現在のForensic Auditorペルソナ）
  性質: 非決定的だが、有用な参考情報
  制約: 最終スコアへの影響は補助的（重みの上限を設定）
```

**重要な転換点:** 以前は「AIが品質を判定し、静的解析が補助する」構造だった。これを逆転させ、**「決定論的指標が品質を判定し、AIが補助する」**構造に改める。

### この設計判断が意味すること

- 「なぜこのアセットは不合格なのか？」と問われた時、「assert文が0個だからです」と答えられる。「AIがそう言ったから」とは言わない。
- AIの評価が日によって変わっても、決定論的指標が安定したスコアの基盤を提供する。
- 将来的にAIモデルを変更しても（GPT→Claude→Gemini等）、品質基準が揺らがない。

### 品質ゲートの運用原則：透明性と実用性 (Pragmatism) の両立

「論理」を追求する過程で陥りやすい「完璧主義の罠」を回避するため、以下の実用的な運用ルールを適用する。

1. **検証結果の完全開示 (Glassbox Verification)**:
   - 品質ゲートでリジェクトされた際、単にスコアを返すのではなく、**各エバリュエータの内訳、標準出力(stdout)、エラー出力(stderr)、およびスタックトレース(Traceback)をすべて開示**する。
   - 「なぜ落ちたのか」をユーザーが1秒で理解できる状態を維持する。

2. **段階的な品質管理 (Gradual Validation)**:
   - **Verified (Score 70+)**: テストコードが必須であり、ランタイム実行にパスした高品質資産。
   - **Unverified (Score 40)**: テストコードが未実装、あるいは実行をスキップした資産。概要欄に `[AI-DRAFT]` を含むことで、この状態での保存を許容する。
   - これにより、「まずは動くロジックを溜める」という初動の摩擦を最小化し、後から非同期に品質を高めていくフローを実現する。

3. **診断メッセージの中立性**:
   - タイムアウト発生時、一律で「無限ループ」と決めつけず、「環境構築のオーバーヘッド」や「ライブラリのロード遅延」の可能性を等しく提示する。
   - ユーザーに「自分のコードが悪い」という根拠なき罪悪感を与えない。

## 🏛️ 補足：エージェント賢者パラドックスと「整備士」の哲学

### 問い：エージェントが賢いなら、なぜこれほど厳しい検品が必要なのか？

エージェント（Antigravity等）が「自分でロジックを直せるほど賢い」のであれば、サーバー側での厳格なバリデーションやスコアリングは無駄な摩擦（Friction）に思えるかもしれません。しかし、LogicHiveはこのパラドックスに対して明確な回答を持っています。

**「エージェントが賢い操縦士だからこそ、渡される部品（VLU）は100%の純度を持つ『鋼』でなければならない。」**

1.  **知能の浪費を防ぐ**: 
    エージェントが「このコードは本当に動くのか？」を確認したり、微細なバグを修正したりすることに知能のリソースを割くのは、高度なソフトウェア設計における「損失」です。
2.  **確実な基盤（Ground Truth）の提供**: 
    エージェントが自信を持って（最速で）プロジェクトを推進するためには、背後に「絶対に嘘をつかない、厳格な整備士」が必要です。整備士が「多分動く」という部品を渡せば、操縦士は疑心暗鬼になり、フライトのパフォーマンスが低下します。
3.  **不純物の排除**: 
    「自分で直せるから、壊れていてもいい」という甘えは、資産の劣化を加速させます。LogicHiveはエージェントのHubris（傲慢、あるいは過信）を、決定論的な事実（AST、実行結果）で律し続け、バグの混入を未然に防ぎます。

私たちは、エージェントを信じないから検品するのではありません。**エージェントの知能を「修正」という低次の作業から解放し、より高次な「創造」へと集中させるため**に、冷徹で厳格な門番（Gatekeeper）であり続けるのです。
